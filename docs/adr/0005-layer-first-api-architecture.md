# Use Layer-First Architecture for the API App

The API app will grow into PlaylistWizard's backend boundary for playlist operations, auth-backed Account access, queue processing, and future WebSocket job updates. We will organize `apps/api` by top-level layers with feature subdirectories rather than by isolated vertical modules: `presentation` for HTTP, queue, and WebSocket entry points; `usecase` for user-visible application usecases; `infrastructure` for DB, auth, Provider APIs, queues, and WebSocket delivery; and `composition` for wiring concrete implementations into usecase ports. Domain models and pure PlaylistWizard business rules should live in `@playlistwizard/core` where they can be shared and tested independently.

Usecase code should be organized around application behavior visible to a user or client, not around reusable internal services. A usecase may coordinate multiple domain concepts, such as Accounts and Playlists, but it should depend on explicit ports for data access, token access, external Provider calls, queueing, transactions, and event publishing. Usecases must not directly import infrastructure implementations, and one usecase should not call another usecase as a shortcut for sharing behavior. Infrastructure owns lower-level implementation helpers such as repository reads, while composition is the place where infrastructure implementations are constructed and connected to usecase ports. This keeps usecase dependencies explicit without introducing the indirection and cross-module friction of strict vertical slices.

Playlist Action remains the domain concept for a user's requested change to Playlists. Playlist Action Jobs, Steps, retry handling, progress tracking, queue processing, and WebSocket notifications are execution concerns coordinated by the usecase layer, not the core domain language.
